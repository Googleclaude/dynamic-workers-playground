# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm install           # install deps
npm start             # vite dev (http://localhost:5173) — runs the Worker locally via @cloudflare/vite-plugin
npm run build         # vite build && wrangler deploy
npm run deploy        # vite build && wrangler deploy
npm run types         # regenerate env.d.ts from wrangler.jsonc
```

There is no test runner, linter, or formatter configured. TypeScript is `noEmit`; type-checking happens implicitly via Vite/Wrangler builds.

The dev server runs the Worker (not just the client) — `/api/*` is routed to `src/server.ts` and everything else is served as static assets per `wrangler.jsonc`'s `run_worker_first` rule.

## Architecture

This is a **Cloudflare Worker that runs other Cloudflare Workers**. The host Worker bundles user-supplied source files at runtime and executes them inside an isolated dynamic worker via the `worker_loaders` binding. Understanding the data flow across `server.ts`, `logging.ts`, and the `LOADER`/`LogSession` bindings is the key to working in this repo.

### Request flow for `POST /api/run` (`src/server.ts`)

1. Client posts `{ files, version, pathname?, options? }`.
2. `normalizeFiles` injects a synthetic `package.json` (with inferred `main`) when one is not provided.
3. `createWorkerId` hashes the sorted files + bundle/minify options into a stable `dynamic-workers-playground-worker-<hash>` ID. Identical inputs → same ID → `env.LOADER.get` returns the cached worker (warm start). Different inputs → the loader callback runs and `@cloudflare/worker-bundler`'s `createWorker({ files, bundle, minify })` produces `mainModule` + `modules`.
4. The loaded worker is configured with a fixed env (`API_KEY`, `DEBUG`, `WORKER_ID`), `globalOutbound: null`, and a single tail Worker — `DynamicWorkerTail` — bound to `props: { workerId }`.
5. `executeWorker` calls `entrypoint.__warmup__?.()` (intentionally non-existent — the catch is the cold-start trigger; do not "fix" this), opens a `LogSession` waiter, fetches the request, then collects logs with a 1s deadline.
6. Response includes `bundleInfo`, the worker's response, captured `logs`, and a `timing` breakdown (`buildTime`/`loadTime`/`runTime`). When the loader cache hits, `bundleInfo` is reported as `(cached)` because `createWorker` was not called.

### Tail-worker → Durable Object log pipeline (`src/logging.ts`)

The dynamic worker has no direct channel back to the host. Logs flow:

```
dynamic worker  →  DynamicWorkerTail.tail(events)  →  LogSession DO (keyed by workerId)  →  LogWaiter  →  /api/run response
```

- `DynamicWorkerTail` is a `WorkerEntrypoint` parameterised by `props.workerId` — that's how each tail instance knows which `LogSession` to write to.
- `LogSession` (Durable Object, declared in `wrangler.jsonc` migration `v1`) holds at most one `LogWaiter` (an `RpcTarget`) per session. `executeWorker` calls `waitForLogs()` *before* dispatching the fetch, then `getLogs(timeoutMs)` after, which resolves either when logs arrive or when the timeout elapses.
- Trace events are split into `request` / `log` / `exception` kinds. Only non-request events are surfaced in the realtime UI feed.
- `runtimeExports` / `contextExports` come from `cloudflare:workers`'s `exports` and the per-request `ctx.exports` — these expose the host Worker's own classes (`LogSession`, `DynamicWorkerTail`) so the loader callback can pass a tail instance into the dynamic worker's config.

### GitHub import (`src/github.ts`)

`POST /api/github` parses URLs of the form `https://github.com/<owner>/<repo>[/tree/<branch>/<path>]`, recursively walks the GitHub Contents API, and returns a `Record<path, contents>` keyed relative to the imported subpath. The client then merges those files into the editor and synthesises a `package.json` if the repo lacked one.

### Client (`src/client/index.tsx`)

Single-file React 19 app using `@cloudflare/kumo` + `@phosphor-icons/react`. Holds editor state, runs `POST /api/run`, displays response/logs/timing/bundle info. The `version` field sent to the server is purely a client-side cache-bust signal; the server's actual cache key is the content hash, so changing `version` without changing files does not force a rebundle.

### Bindings & config (`wrangler.jsonc`)

- `worker_loaders: [{ binding: "LOADER" }]` — the dynamic-worker primitive. Without it the `/api/run` flow cannot work.
- `durable_objects` + `migrations` for `LogSession`. Adding new DO classes requires a new migration tag.
- `assets.run_worker_first: ["/api/*"]` — the Worker only handles `/api/*`; everything else is static.
- `compatibility_date: "2026-01-28"` with `nodejs_compat`.
- `env.d.ts` is generated by `npm run types`; do not hand-edit.

## Conventions

- TypeScript strict mode, `verbatimModuleSyntax`, `allowImportingTsExtensions`, ES2022 modules. Type-only imports must use `import type`.
- The bundled dynamic worker receives a hard-coded env (`API_KEY: "sk-example-key-12345"`, `DEBUG: "true"`, `WORKER_ID`). This is an example value, not a secret — but treat it as the intended interface for example workers.
- Keep `executeWorker`'s warmup pattern intact: the empty-catch on `__warmup__` is load-bohaviour, not a bug.
- When changing the run-request shape, update both `RunRequestBody` in `server.ts` and `RunResult` / the `fetch` call in `src/client/index.tsx`.

## Deployment note

The deployed playground executes arbitrary user-supplied Worker code. Per the README, public deployments should sit behind Cloudflare Access.
